Here's the exact API documentation for how images and videos are fetched and rendered in the player, which you can share with Replit for the mobile app:

How Media (Images, Videos, Avatars) Are Fetched
The CDN Proxy Pattern
All media in the player goes through a Supabase Edge Function proxy to avoid CORS issues and hide the origin server IP. The edge function is:

GET https://oxwhqvsoelqqsblmqkxx.supabase.co/functions/v1/video-generation-proxy
Query Parameters
Param	Required	Description
action	Yes	Always cdn_proxy for student-facing media
job_id	Yes	The video generation job UUID (e.g., 0d01675f-...)
file_path	Yes	Relative path within the job folder
cdn_base_url	Yes	The CDN origin (default: https://server1.simplelecture.com/video)
How the Proxy Works
The edge function constructs the final URL as:

{cdn_base_url}/{job_id}/{file_path}
Example:

https://server1.simplelecture.com/video/0d01675f/images/myopia_diagram.png
It fetches this server-side and returns the response with proper CORS headers, content-type detection, and range request support (for video streaming).

Full Example URLs
Image:

https://oxwhqvsoelqqsblmqkxx.supabase.co/functions/v1/video-generation-proxy?action=cdn_proxy&job_id=0d01675f&file_path=images/topic_3_image_0.png&cdn_base_url=https://server1.simplelecture.com/video
Avatar Video:

https://oxwhqvsoelqqsblmqkxx.supabase.co/functions/v1/video-generation-proxy?action=cdn_proxy&job_id=0d01675f&file_path=avatars/section_3_avatar.mp4&cdn_base_url=https://server1.simplelecture.com/video
Beat/Manim Video:

https://oxwhqvsoelqqsblmqkxx.supabase.co/functions/v1/video-generation-proxy?action=cdn_proxy&job_id=0d01675f&file_path=videos/topic_3_beat_0.mp4&cdn_base_url=https://server1.simplelecture.com/video
Audio:

https://oxwhqvsoelqqsblmqkxx.supabase.co/functions/v1/video-generation-proxy?action=cdn_proxy&job_id=0d01675f&file_path=audio/3.mp3&cdn_base_url=https://server1.simplelecture.com/video
File Path Conventions
Media Type	Path Pattern	Example
Images	images/{image_id}.png	images/topic_3_image_0.png
Avatar (English)	avatars/section_{id}_avatar.mp4	avatars/section_3_avatar.mp4
Avatar (Other lang)	avatars/{language}/section_{id}_avatar.mp4	avatars/hindi/section_3_avatar.mp4
Beat/Manim videos	videos/{video_name}.mp4	videos/topic_3_beat_0.mp4
Audio narration	audio/{section_id}.mp3	audio/3.mp3
Note: Images are always normalized to .png extension regardless of original format.

For Mobile App (Direct Access)
Since mobile apps don't have CORS restrictions, you can skip the proxy and fetch directly:

https://server1.simplelecture.com/video/{job_id}/{file_path}
Example:

https://server1.simplelecture.com/video/0d01675f/images/topic_3_image_0.png
This is simpler and faster for native mobile apps. The proxy is only needed in browsers for CORS.

Where the job_id Comes From
The job_id is stored in the database table video_generation_jobs. When a student opens a topic/chapter, the app queries:


SELECT id, presentation_json FROM video_generation_jobs 
WHERE topic_id = '{topic_id}' AND status = 'completed'
The presentation_json column contains the full slide data including image_id references, avatar paths, and beat video paths that map to the file path patterns above.

Rendering in the UI
Images are rendered as simple <img> tags:


<img src={getCdnMediaUrl(jobId, getImagePath(beat.image_id), cdnBaseUrl)} alt="" />
Videos use standard HTML5 <video> elements with the same URL pattern. The avatar uses WebGL chroma-key rendering on a <canvas> overlaid on the video.

Here's the exact mechanism explained for your Replit mobile implementation:

How Content (Images/Text) Changes Over Time
1. Data Source: presentation_json
Each section in the JSON has two key arrays:

narration.segments[] — array of timed text chunks the avatar speaks. Each has segment_id, duration_seconds, and text.
visual_beats[] — array of content items (images, bullets, text). Each beat has a beat_id, visual_type (text/bullet_list/image/latex), display_text, image_id, and a segment_id linking it to a narration segment.
2. Timing Map (Step 1 on mount)
On section load, a cumulative timing map is built from narration.segments:

Segment 0: duration 8s  → startTime=0,  endTime=8
Segment 1: duration 12s → startTime=8,  endTime=20
Segment 2: duration 10s → startTime=20, endTime=30
This is done by buildSegmentTimingMap() in timingUtils.ts.

3. Time Source: Avatar Video
The avatar video (<video> element) is the master clock. On every timeupdate event (~4x/sec), the player:

Gets video.currentTime
Normalizes it: if the avatar video duration differs from JSON total duration, it maps proportionally:
normalizedTime = (video.currentTime / avatarDuration) * jsonDuration
Calls updateState(normalizedTime)
4. Beat Reveal Logic (the core mechanism)
updateState() calls getRevealedBeats(time) which uses 3 strategies in priority order:

Priority	Strategy	How it works
1	beat.start_time	If the beat has an explicit start_time field, reveal when currentTime >= start_time
2	beat.segment_id	Match beat's segment_id to the timing map. Reveal when currentTime >= segment.startTime
3	Even distribution	No ID? Spread beats evenly across 70% of total duration: beatStartTime = beatIndex * (duration*0.7 / totalBeats)
Result: An array of indices like [0, 1, 2] meaning beats 0, 1, and 2 are visible.

5. Display Directives (show/hide layers)
Each narration segment can have display_directives:


{
  "visual_layer": "show",  // show the beat video/image
  "text_layer": "show"     // show the text/bullets
}
The player checks the current segment's directives to toggle between showing the text content vs. showing a full-screen video/image layer.

6. Rendering
The ContentSection component receives revealedIndices and renders only beats whose index is in that array. The latest revealed beat is the "active" one (highlighted), older ones stay visible but dimmed. On mobile, only the single latest beat is shown (single-bullet pattern).

7. Section Transitions
When the avatar video fires ended, the player:

Sets isPlaying = false
Calls onSectionEnd() which advances to the next section
Resets revealedBeatIndices to [] and currentTime to 0
Loads the new section's avatar video and rebuilds the timing map
Summary for Mobile Implementation
Avatar video timeupdate
  → normalize time (avatar duration → JSON duration)
  → find current segment index (loop timing map)
  → compute revealed beats (segment_id matching)
  → update UI: show only revealed beats, apply display directives
  → on video ended: advance to next section, reset state
The key data fields from presentation_json you need per section:

narration.segments[].segment_id + duration_seconds → build timing map
visual_beats[].segment_id → link beats to segments
visual_beats[].visual_type + display_text + image_id → render content
narration.segments[].display_directives → show/hide layers
